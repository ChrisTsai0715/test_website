# NotifyHub API - LLM-Optimized Documentation

> This document is an LLM-friendly conversion of the NotifyHub API documentation, structured to help AI assistants guide users in utilizing the NotifyHub service effectively.
> 
> Original Documentation: https://docs.notifyhub.example.com
> 
> Last Updated: March 2026

---

## Core Principles

When helping users implement NotifyHub APIs, follow these guidelines:

0. **API Token Required**: All requests require an `api-token` header for authentication. For testing, use `YOUR_API_TOKEN`. For production, users must contact support@notifyhub.example.com to obtain their own token.
1. **Security First**: Never expose API tokens in client-side code. Store them securely in environment variables or backend services.
2. **Base URL**: The test environment endpoint is `https://api.notifyhub.example.com`. Production URLs should be obtained from the NotifyHub team.
3. **Timeout Configuration**: All APIs support `wait_for_timeout` (page rendering timeout, max 60s).
4. **Response Format**: All responses follow a unified JSON structure with `success`, `result`, `context`, and `errors` fields.
5. **Error Handling**: Check the `success` field first. If false, examine the `errors` array for details. HTTP status codes indicate different error types (400: parameter error, 422: request failed, 429: rate limited).
---

## Overview of All NotifyHub APIs

NotifyHub provides four main categories of APIs:

### Reader APIs
- **content**: Fetch the complete HTML content of a webpage after rendering
- **markdown**: Convert webpage content to LLM-friendly markdown format
- **screenshot**: Capture a screenshot of a webpage
- **links**: Extract all links from a webpage
- **snapshot**: Get a complete webpage snapshot (including screenshot and content)
- **pdf**: Convert a webpage to PDF format

### AI-Powered APIs
- **json**: Use AI to extract structured data from webpages based on prompts or schemas (BETA)

### Search APIs
- **serp**: Retrieve search engine results page for a given keyword

### MCP Integration
- **Reader MCP**: Access webpage content through Model Context Protocol for AI applications
- **Browser Operation MCP** (NEW): Full browser automation capabilities through MCP, enabling AI agents to navigate, click, type, and interact with webpages like a human user

- The `wait_for_timeout` parameter controls page rendering time, not the total API response time

### 2. Markdown API - Convert Webpage to Markdown

**Endpoint:** `POST https://api.notifyhub.example.com/v1/markdown`

**Purpose:** Access a specified URL and extract content in LLM-friendly markdown format.

**Best for:**
- Converting web content for AI/LLM consumption

**Method:** POST

**Authorization:** Header-based API token

**Headers:**
- **api-token** (required): Your API authentication token
- **Content-Type** (required): `application/json`

**Request Body Schema:**
```json

    "type": "string",
    "required": true,
    "description": "The target URL to access and convert to markdown"
  },
  "markdown_options": {
    "type": "object",
    "required": false,

      "md_hr": {
        "type": "string",

        "type": "string",

      },
      "md_em_delimiter": {
        "type": "string",
        "required": false,
        "default": "_",
        "description": "Define markdown emphasis delimiter",
        "options": ["_", "*"]

      "md_strong_delimiter": {
        "type": "string",
```

**Response Fields:**
- **result** (string): The webpage content converted to markdown format
- Other fields same as Content API

**Notes:**
- The markdown conversion automatically removes unnecessary HTML elements
- Preserves important structural elements like headings, lists, links, and text formatting
- Ideal for feeding web content into LLMs or text analysis tools

- The response is the image file itself (PNG or JPEG), ready to be saved directly

**Notes:**
- Unlike text-based NotifyHub APIs, the Screenshot API returns **binary image data directly**, not JSON
- Simply save the response body to an image file (`.png` or `.jpeg` depending on your `screenshot_options.type`)
- Useful for visual regression testing and webpage monitoring
- The screenshot captures the full rendered page as it appears in the browser

## Error Handling

NotifyHub API uses standard HTTP status codes to indicate request outcomes. All responses follow a unified JSON format.

### HTTP Status Codes

#### 1. HTTP 200 - Success

**Example Response:**
```json
{
  "success": true,
  "result": "...",
  "context": {
    "status_code": 200,
    "title": "Example Domain"
  },
  "errors": []
}
```

**Description:** Request completed successfully. The `success` field is `true` and the `result` field contains the requested data.

---

#### 2. HTTP 400 - Parameter Error

**Example Response:**
```json
{
  "success": false,
  "result": "",
  "context": {},
  "errors": ["Parameter error: url field cannot be empty"]
}
```

**Common Causes:**
- Missing required parameters (e.g., `url`)
- Invalid parameter types
- Parameter values out of valid range

**Resolution:** Check the `errors` array for specific parameter issues and correct your request.

---

#### 3. HTTP 422 - Request Failed

**Example Response:**
```json
{
  "success": false,
  "result": "",
  "context": {},
  "errors": ["Webpage access failed: connection timeout"]
}
```

**Common Causes:**
- Target URL is unreachable
- Network timeout
- Target server returned an error
- Page rendering failed

**Resolution:** 
- Verify the target URL is accessible
- Increase `wait_for_timeout` if the page is slow to load
- Check if the target website blocks automated access

---

#### 4. HTTP 429 - Rate Limit Exceeded

**Example Response:**
```json
{
  "success": false,
  "result": "",
  "context": {},
  "errors": ["Too many requests, please try again later"]
}
```

**Common Causes:**
- Exceeded rate limit for your API token
- Too many concurrent requests

**Resolution:**
- Implement exponential backoff retry logic
- Reduce request frequency
- Contact NotifyHub team for higher rate limits if needed

## Best Practices

### Performance Optimization
1. **Optimize Timeouts:** Set appropriate `wait_for_timeout` values based on target webpage complexity
2. **Use Markdown API:** When feeding content to LLMs, prefer the markdown endpoint for cleaner, more structured data

### Security
1. **Token Management:** Store API tokens in environment variables, never in source code
2. **Backend Proxy:** Route NotifyHub API calls through your backend to avoid exposing tokens
3. **Rate Limiting:** Implement client-side rate limiting to avoid hitting API limits

### Reliability
1. **Error Handling:** Always check the `success` field before processing results
2. **Retry Logic:** Implement exponential backoff for transient failures (HTTP 422, 429)
3. **Timeout Handling:** Set reasonable timeout values to balance completeness and response time

### Content Processing
1. **Clean HTML:** Use `exclude_tags` to remove unwanted elements like scripts or styles
2. **Wait for Selectors:** Use `wait_for_selector` for precise control over dynamic content loading
3. **Custom Headers:** Set `user_agent` and `extra_http_headers` when accessing sites with specific requirements
